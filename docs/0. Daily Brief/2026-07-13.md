### Content of the Day — APT / Intrusion Profile: Helix (identity-first SharePoint extortion)

**Helix** is a newly named data-extortion crew (surfaced via ReliaQuest, mid-July) that never touches a password and rarely touches malware. Its kill chain is pure identity abuse. It opens with **vishing (T1566.004)** — an operator phones an employee while **impersonating their manager or IT (T1656)**, often spoofing the caller ID — and steers them into a **device-code phishing** flow. The victim is told to visit `microsoft.com/devicelogin` and enter a code the attacker generated; behind the scenes the attacker initiated Microsoft's OAuth Device Authorization Grant, so when the user "approves," the **session token gets issued to the attacker (T1528 / replayed via T1550.001)**. No password is typed, and because it's a legitimate token minted by Microsoft's own IdP, it sails past **Conditional Access** and looks like normal auth traffic to every proxy and mail filter in the environment.

Once holding a token, Helix moves fast for persistence: it **registers a fresh MFA authenticator (T1098.005 – Device Registration)** so it survives the original session. Then it goes to work on **SharePoint/OneDrive** — a period of manual browsing followed by **scripted enumeration and bulk download (T1213.002 → T1567)** staged from a dedicated hosting IP used only for exfil. In one case ReliaQuest documented, SharePoint access with *no recorded auth method* happened six minutes after MFA registration — a device-code token replayed straight in. Unlike ransomware, exfil-only extortion triggers no outage, so victims often don't notice until the demand lands.

The lineage matters more than the brand. Helix's playbook and shared registrar/hosting infrastructure (e.g. phishing domain `oskeysync[.]com` via NICENIC) closely mirror the defunct **BlackFile (UNC6671)** — which fragmented into Pink and Redact after its April 2026 shutdown — and overlaps with **ShinyHunters (UNC6661)**. Vishing → device-code phishing → MFA abuse → SharePoint theft is now *shared tradecraft* across this whole ecosystem, not one group's signature. The single highest-impact defensive control is boring and decisive: **disable device-code authentication** where you don't explicitly need it, and restrict sensitive SaaS to managed endpoints.

**Lab/cert tie-in:** this is Azure/M365 red-team territory rather than classic on-prem OSCP, but device-code phishing shows up increasingly in CRTP/CPTS-adjacent cloud assessments and is worth having hands-on. In an **authorized lab tenant**, `TokenTactics`/`TokenTacticsV2` drives the device-code flow (`Get-AzureToken -Client Graph`), then `GraphRunner` (`Invoke-SearchSharePointAndOneDrive`) or `roadrecon` mimics exactly the enumerate-and-pillage phase Helix runs — a clean way to *feel* why disabling the device-code grant closes the whole chain.

### One Thing to Try

In an **authorized test tenant**, run the device-code grant end to end with `Get-AzureToken -Client Graph` (TokenTacticsV2), approve it as the "victim," then replay the token with `GraphRunner`'s `Invoke-SearchSharePointAndOneDrive` — watch the SharePoint access land with no password and no fresh MFA prompt.
