### Content of the Day — Linux Privilege Escalation: File Capabilities

Everyone drills SUID binaries; far fewer instinctively check **file capabilities**, and they're a frequent, quietly-planted privesc on HTB and Proving Grounds boxes. Capabilities split root's monolithic power into ~40 discrete units (see `capabilities(7)`), and a binary can carry a subset in its extended attributes so it runs with just those privileges — no SUID bit required. The mechanics that matter: a capability lands in a binary's **permitted/effective/inheritable** sets, and the flag suffix is everything. `cap_setuid+ep` means the cap is both permitted *and* effective on exec — instantly usable. `+ei` (inheritable/effective, no permitted) or `=ep` variants change whether you can actually wield it, so read the flags, don't just grep the cap name. Enumerate with `getcap -r / 2>/dev/null` — that recursive scan is the whole game, and it's the step people skip.

The payoffs are tiered. **`cap_setuid`** is game over: on an interpreter you get an immediate root shell — `python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'` if `python3` carries it, and the same idea works for `perl`, `ruby`, or `node`. **`cap_dac_read_search`** is the nuanced one worth internalizing: it bypasses read permission checks but **not** write, so you don't get to edit `/etc/passwd` — instead you *exfiltrate* `/etc/shadow` and crack offline, or read root's SSH keys. **`cap_dac_override`** *does* bypass write checks, which reopens the classic `/etc/passwd` or sudoers overwrite. **`cap_sys_admin`** is near-omnipotent (mount/namespace tricks), and other GTFOBins-listed carriers like `tar`, `gdb`, `openssl`, or `vim` become privesc primitives the instant they hold `cap_setuid`. This maps to **MITRE ATT&CK T1548 (Abuse Elevation Control Mechanism)** — adjacent to the T1548.001 Setuid/Setgid sub-technique — with the recursive `getcap` sweep sitting under **T1083 (File and Directory Discovery)**.

Two operator notes. First, `getcap` isn't always present on a stripped target — if it's missing you can fall back to reading the `security.capability` xattr directly (`getfattr -n security.capability -m - -R / 2>/dev/null`) or just push a static `getcap` binary. Second, capabilities are a legitimate admin convention (it's how `ping` works without SUID on modern distros), so the *defensive* framing is exactly your enumeration checklist: any capability on an interpreter, archiver, debugger, or editor is a finding. **linpeas** flags these automatically and highlights the dangerous ones in red, but knowing the manual path means you're not blind when you can't drop a script. OSCP/CPTS tie-in: this is a standard "I have a shell, now root me" step — bake `getcap -r / 2>/dev/null` into your Linux privesc muscle memory right next to `find / -perm -4000`.

### One Thing to Try

Root a lab box the fast way: `getcap -r / 2>/dev/null` — then if anything shows `cap_setuid`, pop a shell with `<that-binary> -c 'import os; os.setuid(0); os.system("/bin/sh")'` (adjust syntax per interpreter, and check [GTFOBins → Capabilities](https://gtfobins.github.io/#+capabilities) for the exact one-liner).
