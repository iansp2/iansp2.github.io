### Content of the Day — Linux Privilege Escalation: Dynamic Linker Hijacking

You've drilled SUID binaries and (last week) file capabilities. The dynamic linker is the adjacent surface most people under-hit. When a dynamically-linked binary runs, `ld.so` resolves its shared objects at load time, and that resolution is influenceable — sometimes by environment, sometimes by attributes baked into the ELF itself. Two distinct vectors matter, and the nuance between them is exactly what separates a cheat-sheet answer from actually landing the shell. This maps to **MITRE T1574.006 (Hijack Execution Flow: Dynamic Linker Hijacking)**, with the sudo path overlapping **T1548.003 (Abuse Elevation Control Mechanism: Sudo and Sudo Caching)**.

Vector one is `LD_PRELOAD` through a permissive sudo policy. Normally the linker strips `LD_PRELOAD`/`LD_LIBRARY_PATH` in secure-execution mode (`AT_SECURE`, set for setuid/setgid binaries) — that's why you can't just preload a `.so` into a SUID root binary. But `sudo` runs the child fresh, and if `sudoers` carries `Defaults env_keep += "LD_PRELOAD"` (or env_reset is weakened), the variable survives into a root-context child *even for a harmless allowed command*. So a one-line `sudo -l` entry you'd otherwise ignore becomes root. Build a shared object with a constructor and preload it:

```c
// x.c
#include <stdlib.h>
void _init(){ unsetenv("LD_PRELOAD"); setgid(0); setuid(0); system("/bin/bash -p"); }
```
```bash
gcc -fPIC -shared -nostartfiles -o /tmp/x.so x.c
sudo LD_PRELOAD=/tmp/x.so <any-allowed-binary>   # -> root
```

Vector two is the secure-execution-*safe* one, and the reason this topic is worth a whole entry: **RPATH/RUNPATH**. Those are library search paths embedded in the ELF, and `AT_SECURE` does **not** strip them the way it strips `LD_*` env vars. So if a SUID-root binary was linked with a RUNPATH pointing at a directory you can write to (sloppy `-Wl,-rpath` at build time, or a `$ORIGIN`-relative path under your control), you drop a malicious `.so` matching the expected SONAME there and the linker loads *your* code with root privileges — no environment tricks, no sudo needed. Enumerate it:

```bash
find / -perm -4000 -type f 2>/dev/null                        # SUID hunt
readelf -d ./target | grep -E 'RPATH|RUNPATH|NEEDED'          # embedded paths + deps
ldd ./target                                                   # what resolves, and from where
```

If a RUNPATH dir is group/world-writable, compile a `.so` exporting the missing symbol (or just a constructor) named to match a `NEEDED` entry, drop it in, and re-run the binary. LinPEAS flags both the `env_keep` policy and writable-library-path cases, but knowing *why* AT_SECURE blocks one vector and not the other is what lets you triage a real box fast. **OSCP/lab tie-in:** the sudo-`LD_PRELOAD` path is a recurring Proving Grounds Practice privesc, and the writable-RUNPATH-on-custom-SUID-binary variant is a classic on HTB boxes that ship a bespoke SUID helper — always `readelf -d` the custom binary before assuming it's a straight GTFOBins job.

### One Thing to Try

Run `sudo -l`; if you see `env_keep+=LD_PRELOAD`, drop the constructor `.so` above and `sudo LD_PRELOAD=/tmp/x.so <allowed-cmd>`. No sudo? Sweep for the RUNPATH variant: `find / -perm -4000 -type f 2>/dev/null -exec readelf -d {} \; 2>/dev/null | grep -i runpath`.
