### Content of the Day

Web Exploitation: Trusted reverse-proxy header spoofing (`X-WEBAUTH-USER` and friends)

The Gitea bug up top is a perfect specimen of a bug class you'll hit far more often than any single CVE: **the app trusts an identity header that it should only accept from a trusted upstream.** The pattern is everywhere. An app is designed to sit behind an authenticating reverse proxy (nginx, Apache, an SSO gateway). The proxy validates the user, then injects a header — `X-WEBAUTH-USER`, `X-Remote-User`, `X-Forwarded-User`, `X-Authenticated-User`, `X-Forwarded-For`, `X-Real-IP` — telling the backend "this request is already authenticated as `bob`." The backend trusts it blindly. The whole security model depends on the backend *only* accepting that header from the proxy's IP. When that allowlist is missing or wildcarded (`REVERSE_PROXY_TRUSTED_PROXIES = *` in Gitea's case), anyone who can talk to the backend directly forges the header and becomes whoever they want. Maps to **T1190 (Exploit Public-Facing Application)** for the entry, resolving to **T1078 (Valid Accounts)** since you end up as a legitimately-authenticated principal — often admin.

Exploiting it is genuinely a one-liner. Against a lab Gitea: `curl -H "X-WEBAUTH-USER: gitea_admin" http://target:3000/api/v1/user` — if you get back the admin's profile instead of a 401, you own it. From there it's read every private repo, plant an SSH key for persistence, tamper with webhooks to exfil future pushes, and loot committed secrets (API keys, DB creds, deploy tokens). The enumeration move that finds this without a known CVE: take any endpoint that requires auth, then replay it while spraying identity headers. Burp's **Param Miner** ("guess headers") or a quick `ffuf`/Intruder run against a header wordlist (`X-Remote-User`, `X-Forwarded-User`, `X-WEBAUTH-USER`, `X-Original-URL`, `X-Rewrite-URL`) surfaces it fast. Two adjacent tricks in the same family worth keeping loaded: `X-Forwarded-For: 127.0.0.1` to defeat "admin panel is localhost-only" checks, and `X-Original-URL` / `X-Rewrite-URL` for proxy path-based ACL bypass.

The reason this is high-value for OSCP/CPTS is that it's an *authentication* bypass reachable with zero exploit code — pure request manipulation, no payload, no shell needed to get the initial win. On HTB/PG you'll see it wherever a box runs an internal app "protected" by a front-end proxy: reach the backend port directly (often a different port you found in your nmap sweep) and the proxy's identity assertions become attacker-controlled. It's also a clean example of why you always port-scan all 65535 and probe the raw upstream, not just the polished front door.

### One Thing to Try
Header-spray any auth-gated endpoint on a lab box: `ffuf -w headers.txt -u http://TARGET:PORT/admin -H "FUZZ: admin" -mc 200` (seed `headers.txt` with `X-WEBAUTH-USER`, `X-Remote-User`, `X-Forwarded-User`, `X-Forwarded-For`) — watch for the 200 that shouldn't be there.
